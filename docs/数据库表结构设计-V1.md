# 数据库表结构设计 V1

## 1. 文档目标

- 基于 [软件需求规格说明书-V1.md](file:///c:/Users/15034/GolandProjects/final-exam-savior/docs/%E8%BD%AF%E4%BB%B6%E9%9C%80%E6%B1%82%E8%A7%84%E6%A0%BC%E8%AF%B4%E6%98%8E%E4%B9%A6-V1.md) 输出严格贴合需求、尽量满足三范式的数据库设计。
- 目标是让后端开发可以据此完成建表、实体设计和索引设计。
- 本设计以 `MySQL 8.x` 为基准，字符集建议使用 `utf8mb4`，排序规则建议使用 `utf8mb4_0900_ai_ci`。

## 2. 复核结论

- 原版本中 `email_verification_codes` 与现有实现冲突，因为邮箱验证码明确存储于 `Redis`，不应进入数据库表设计。
- 原版本中 `learning_files` 同时承载了：
- 文件主数据
- 预览转换结果
- 最新生成记录状态
- 这会让单表职责过重，不利于严格按三范式拆分。
- 本次重构后，数据库只保留真正需要持久化、且在需求中明确属于数据库职责的数据。
- 验证码、发码频控、登录失败计数这类高频短时状态，建议统一走 `Redis`，不纳入数据库主表设计。

## 3. 设计原则

- 主键统一使用 `BIGINT UNSIGNED`。
- 时间字段统一使用 `DATETIME(3)`。
- 业务状态统一使用 `VARCHAR(32)` 或 `VARCHAR(64)` 枚举值，提升可读性。
- 每张表只描述一个核心实体或一个明确关系，避免把“主数据 + 流程状态 + 历史明细”混在同一张表中。
- 文件主数据、最新生成结果、历史任务、通知记录分开建模。
- 第一版不启用数据库物理外键约束，仅保留关联字段和索引，由应用层保证数据一致性，便于开发和测试。
- 对象存储资源第一版仅保存 `object_url`，严格贴合需求文档。
- 文件删除采用物理删除，因此不设计通用软删除字段。
- 第一版不保留邀请码使用记录，因此不设计邀请码使用流水表。
- 第一版不保留文件删除日志，因此不设计文件删除日志表。

## 4. 不进入数据库的状态

以下内容建议明确由 `Redis` 承载，不在 MySQL 里建表：

- 注册验证码
- 忘记密码验证码
- 验证码 60 秒 / 5 分钟 / 3 小时 / 24 小时频控计数
- 登录失败风控计数
- 验证码错误次数的短时状态

**建议的 Redis Key 方向**

- `email:code:{bizType}:{email}`
- `email:send_limit:{email}:{window}`
- `login:fail:{email}:{window}`

## 5. 表清单

- `users`：用户主表
- `user_sessions`：用户登录会话表，第一版以 MySQL 为主存储
- `invite_codes`：邀请码表
- `file_categories`：文件分类表
- `learning_files`：学习资料主表
- `file_preview_records`：文件最新预览记录表
- `file_generate_records`：文件最新生成记录表
- `file_generate_record_items`：最新生成结果项表
- `generate_tasks`：生成总任务历史表
- `generate_task_items`：生成子任务历史表
- `preview_conversion_tasks`：预览转换任务历史表
- `task_retry_logs`：任务重试日志表
- `system_notifications`：站内通知表

## 6. 表结构设计

### 6.1 `users`

**用途**

- 存储普通用户与管理员账号。
- 支撑登录、权限、账号启用禁用、修改密码、按邮箱搜索、展示注册时间。

**字段设计**

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | `BIGINT UNSIGNED` | 是 | 主键 |
| `email` | `VARCHAR(128)` | 是 | 用户邮箱，唯一 |
| `password_hash` | `VARCHAR(255)` | 是 | 强哈希加盐密码摘要 |
| `role` | `VARCHAR(32)` | 是 | `ADMIN` / `USER` |
| `status` | `VARCHAR(32)` | 是 | `ENABLED` / `DISABLED` |
| `registered_at` | `DATETIME(3)` | 是 | 注册时间 |
| `last_login_at` | `DATETIME(3)` | 否 | 最后登录时间 |
| `disabled_at` | `DATETIME(3)` | 否 | 禁用时间 |
| `disabled_by` | `BIGINT UNSIGNED` | 否 | 禁用操作者用户 ID |
| `remark` | `VARCHAR(255)` | 否 | 管理备注 |
| `created_at` | `DATETIME(3)` | 是 | 创建时间 |
| `updated_at` | `DATETIME(3)` | 是 | 更新时间 |

**索引设计**

- 主键：`PK(id)`
- 唯一索引：`UK_users_email(email)`
- 普通索引：`IDX_users_role(role)`
- 普通索引：`IDX_users_status(status)`
- 普通索引：`IDX_users_registered_at(registered_at)`

### 6.2 `user_sessions`

**用途**

- 支撑主动退出、改密失效、忘记密码失效、禁用账号失效。
- 第一版以 MySQL 为会话主存储，不采用 Redis 作为唯一会话存储。

**字段设计**

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | `BIGINT UNSIGNED` | 是 | 主键 |
| `user_id` | `BIGINT UNSIGNED` | 是 | 用户 ID |
| `session_token` | `VARCHAR(128)` | 是 | 会话标识，唯一 |
| `status` | `VARCHAR(32)` | 是 | `ACTIVE` / `INVALIDATED` / `EXPIRED` |
| `login_ip` | `VARCHAR(64)` | 否 | 登录 IP |
| `user_agent` | `VARCHAR(512)` | 否 | 浏览器或设备信息 |
| `issued_at` | `DATETIME(3)` | 是 | 签发时间 |
| `expires_at` | `DATETIME(3)` | 是 | 过期时间 |
| `invalidated_at` | `DATETIME(3)` | 否 | 失效时间 |
| `invalid_reason` | `VARCHAR(64)` | 否 | `LOGOUT` / `CHANGE_PASSWORD` / `RESET_PASSWORD` / `ACCOUNT_DISABLED` |
| `created_at` | `DATETIME(3)` | 是 | 创建时间 |
| `updated_at` | `DATETIME(3)` | 是 | 更新时间 |

**索引设计**

- 主键：`PK(id)`
- 唯一索引：`UK_user_sessions_token(session_token)`
- 普通索引：`IDX_user_sessions_user_id(user_id)`
- 普通索引：`IDX_user_sessions_status_expires(status, expires_at)`

**用途**

- 存储邀请码主数据。
- 支撑随机/自定义、批量生成、剩余次数扣减、备注修改、停用删除、xlsx 导出。

**字段设计**

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |

| `batch_no` | `VARCHAR(64)` | 否 | 批次号 |
| `total_quota` | `INT UNSIGNED` | 是 | 总可用次数 |
| `remaining_quota` | `INT UNSIGNED` | 是 | 剩余可用次数 |
| `status` | `VARCHAR(32)` | 是 | `ACTIVE` / `DISABLED` |
| `remark` | `VARCHAR(255)` | 否 | 管理员备注 |
| `created_by` | `BIGINT UNSIGNED` | 是 | 创建管理员 ID |
| `created_at` | `DATETIME(3)` | 是 | 创建时间 |
| `updated_at` | `DATETIME(3)` | 是 | 更新时间 |

**索引设计**

- 主键：`PK(id)`
- 唯一索引：`UK_invite_codes_code(code)`
- 普通索引：`IDX_invite_codes_batch_no(batch_no)`
- 普通索引：`IDX_invite_codes_status(status)`
- 普通索引：`IDX_invite_codes_created_by(created_by)`

**说明**

- 邀请码永不过期，不设计 `expires_at`。
- 第一版不保留邀请码使用记录，注册成功时仅扣减 `remaining_quota`。
- 当 `remaining_quota = 0` 或管理员停用时，记录被物理删除。

## 9. 关键业务规则落表说明

### 9.1 重复文件覆盖

- `learning_files.source_file_hash` 唯一约束用于识别相同文件内容。
- 上传相同哈希文件时，不新增主文件记录，而是更新 `learning_files` 当前信息。
- 最新生成结果同步覆盖到：
- `file_generate_records`
- `file_generate_record_items`
- `file_preview_records`

### 9.2 数据库只保留最近一次生成记录

- 通过 `file_generate_records` + `file_generate_record_items` 保存“当前最新生成结果”。
- 通过 `generate_tasks` + `generate_task_items` 保存“30 天内任务历史”。
- 当前结果表严格只保留最新一份记录，不保留多版本。
- 当前结果与历史任务分离，既满足需求，又符合三范式。

### 9.3 文件删除后任务和通知仍保留

- `learning_files`、`file_preview_records`、`file_generate_records`、`file_generate_record_items` 在文件删除时一并物理删除。
- `generate_tasks`、`generate_task_items`、`preview_conversion_tasks`、`system_notifications` 继续保留。
- 因此历史表中必须保留必要快照字段，例如：
- `file_snapshot_name`
- `file_snapshot_hash`
- `file_deleted_snapshot`
- `target_snapshot_name`

### 9.4 邀请码不保留使用记录

- 本设计不含邀请码使用流水表。
- 邀请码次数变化通过更新 `remaining_quota` 实现。
- 当次数耗尽或停用时直接删除记录。

### 9.5 通知 30 天懒清理

- `system_notifications.expires_at` 默认等于创建时间 + 30 天。
- 用户访问通知列表或通知详情时执行懒清理。

### 9.6 Redis 与数据库职责分离

- 验证码、验证码错误次数、发码频控、登录失败计数不落库。
- 数据库只负责持久化用户、会话、文件、任务、通知等长期状态。

### 9.7 任务相关历史统一清理

- `generate_tasks`
- `generate_task_items`
- `preview_conversion_tasks`
- `task_retry_logs`
- 第一版统一按 30 天一起清理，不做额外延长保留。

### 9.8 无物理外键约束的实现要求

- 所有关联 ID 的存在性校验由应用层负责。
- 删除文件、分类、用户等操作时，由服务层显式控制级联逻辑。
- 涉及多表写入的流程必须使用事务，避免出现脏数据和孤儿数据。
- 关键关联字段必须保留索引，避免无外键约束后查询性能退化。

## 11. 为什么这版更符合三范式

- `learning_files` 只描述文件主数据，不再混入预览状态和生成结果。
- `file_preview_records` 只描述文件当前预览状态，不混入任务历史。
- `file_generate_records` 只描述文件当前最新生成总状态，不再重复保存上传人和上传时间。
- `file_generate_record_items` 用“行模型”代替原先 `question/knowledge/extended` 三组并列字段，避免一张表内同构字段重复。
- `generate_tasks` / `generate_task_items` 只负责历史任务，不再承担“当前文件状态”职责。
- `system_notifications` 只负责消息中心，不与任务表混存消息字段。

## 13. 推荐结论

- 第一版推荐采用“主数据表 + 当前状态表 + 历史任务表 + 会话表 + 通知表”的结构。
- 数据库只保留长期状态，短时高频状态交由 `Redis`。
- 对象存储字段第一版仅保留 URL，严格贴合需求文档。
- 当前版本相比旧版更贴近需求，也更适合按三范式组织后端数据模型。
