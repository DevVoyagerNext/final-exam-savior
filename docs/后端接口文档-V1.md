# 后端接口文档 V1

## 1. 文档说明

- 文档目标：基于 [软件需求规格说明书-V1.md](file:///c:/Users/15034/GolandProjects/final-exam-savior/docs/%E8%BD%AF%E4%BB%B6%E9%9C%80%E6%B1%82%E8%A7%84%E6%A0%BC%E8%AF%B4%E6%98%8E%E4%B9%A6-V1.md) 和 [数据库表结构设计-V1.md](file:///c:/Users/15034/GolandProjects/final-exam-savior/docs/%E6%95%B0%E6%8D%AE%E5%BA%93%E8%A1%A8%E7%BB%93%E6%9E%84%E8%AE%BE%E8%AE%A1-V1.md) 输出第一版后端接口方案。
- 接口风格：`RESTful + JSON`
- 接口前缀：`/api/v1`
- 认证方式：`Bearer Token`
- 文件上传方式：`multipart/form-data`
- 时间格式：统一使用 `ISO 8601` 或后端统一约定的 `yyyy-MM-dd HH:mm:ss.SSS`

## 2. 通用约定

### 2.1 通用请求头

- `Authorization: Bearer {token}`
- `Content-Type: application/json`

### 2.2 通用响应结构

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "requestId": "2c3a7a4e7f7d4c15a0d6f6b8f5f0a001"
}
```

### 2.3 响应码建议

- `0`：成功
- `40001`：参数错误
- `40002`：业务校验失败
- `40101`：未登录或登录态失效
- `40301`：无权限
- `40401`：资源不存在
- `40901`：状态冲突
- `42901`：触发频控或风控
- `50001`：系统异常

### 2.4 分页结构

```json
{
  "list": [],
  "pageNo": 1,
  "pageSize": 20,
  "total": 135,
  "totalPages": 7
}
```

### 2.5 权限角色

- `ADMIN`：管理员
- `USER`：普通用户

### 2.6 极验 GT4 接入约定

- 前端业务页面需在页面加载时初始化极验 GT4，脚本地址：`https://static.geetest.com/v4/gt4.js`
- 前端初始化时需传入业务方在极验后台申请的 `captchaId`
- 适用场景：
- `注册前发送邮箱验证码`
- `注册提交前`
- `忘记密码前发送邮箱验证码`
- `登录提交前`
- 前端验证成功后，业务请求需原样携带以下参数给后端：
- `lot_number`
- `captcha_output`
- `pass_token`
- `gen_time`
- `captcha_id`
- 后端收到业务请求后，不信任前端校验结果本身，必须再调用极验二次校验接口：
- `https://gcaptcha4.geetest.com/validate`
- 后端调用极验二次校验时，除前端传入的 5 个字段外，还需按极验官方服务端示例生成并补充：
- `sign_token`
- 只有极验二次校验返回 `status=success` 且 `result=success` 时，后端才允许继续发码、注册或登录
- 若极验校验失败，统一返回业务错误，提示用户重新验证

## 3. 认证与账号接口

### 3.1 发送注册验证码

- 方法：`POST`
- 路径：`/api/v1/auth/register/email-code/send`
- 权限：匿名可访问
- 说明：发送前必须完成极验 GT4 校验；接口收到前端参数后，后端需立即发起极验二次校验；验证码存储于 `Redis`

**请求体**

```json
{
  "email": "123456789@qq.com",
  "lot_number": "4dc3cfc2cdff448cad8d13107198d473",
  "captcha_output": "xxx",
  "pass_token": "xxx",
  "gen_time": "1722672000000",
  "captcha_id": "your-captcha-id"
}
```

**响应 data**

```json
{
  "expireSeconds": 180,
  "nextSendAfterSeconds": 60
}
```

### 3.2 用户注册

- 方法：`POST`
- 路径：`/api/v1/auth/register`
- 权限：匿名可访问
- 说明：注册提交前必须完成极验 GT4 校验；接口收到前端参数后，后端需立即发起极验二次校验；校验通过后才允许校验邀请码、验证码并创建账号；注册成功后直接完成登录，返回单 `token`

**请求体**

```json
{
  "email": "123456789@qq.com",
  "emailCode": "123456",
  "password": "12345678",
  "confirmPassword": "12345678",
  "inviteCode": "OS-2026-001",
  "lot_number": "4dc3cfc2cdff448cad8d13107198d473",
  "captcha_output": "xxx",
  "pass_token": "xxx",
  "gen_time": "1722672000000",
  "captcha_id": "your-captcha-id"
}
```

**响应 data**

```json
{
  "token": "eyJhbGciOiJIUzI1NiJ9.xxx",
  "expireAt": "2026-05-25 20:30:00.000",
  "user": {
    "id": 10001,
    "email": "123456789@qq.com",
    "role": "USER",
    "status": "ENABLED"
  }
}
```

## 13. 关键实现说明

### 13.1 验证码与风控

- 邮箱验证码、验证码频控、登录失败风控计数不落库。
- 这部分状态统一放 `Redis`。
- 极验 GT4 的前端初始化在业务页面加载时完成，不单独提供“初始化接口”。
- 前端业务请求统一携带：
- `lot_number`
- `captcha_output`
- `pass_token`
- `gen_time`
- `captcha_id`
- 后端在注册发码、注册提交、忘记密码发码、登录四个场景中，都必须先调用极验二次校验接口：
- 后端在注册发码、注册提交、忘记密码发码、登录四个场景中，都必须先调用极验二次校验接口：
- `https://gcaptcha4.geetest.com/validate`
- 后端请求极验时使用 `application/x-www-form-urlencoded`，并提交：
- `lot_number`
- `captcha_output`
- `pass_token`
- `gen_time`
- `captcha_id`
- `sign_token`
- 极验二次校验结果处理建议：
- `status=success` 且 `result=success`：继续执行业务逻辑
- `status=success` 且 `result=fail`：返回 `40002`，提示用户重新验证
- `status=error`：返回 `50001` 或映射后的接入错误码，并记录极验错误详情

### 13.2 源文件预览和 HTML 结果预览

- 源文件预览走 `/preview-source`
- HTML 结果预览走 `/preview-result`
- 两套接口都先做权限校验，但后续逻辑不同：
- 源文件预览需要判断文件类型、是否需要转换 PDF、当前转换状态
- HTML 结果预览直接返回已生成 HTML 的临时可访问地址
- 预览与下载接口在鉴权通过后，统一返回 `url + expireAt`
- 第一版临时 URL 有效期固定为 `10 分钟`

### 13.3 重复文件覆盖

- 通过 `source_file_hash` 判断重复文件
- 若哈希重复，不重新生成内容
- 更新文件主信息，并覆盖当前最新生成记录
- 同时新建一条任务记录，任务状态直接标记为成功，并注明复用旧结果

- `generate_task_items`
- `preview_conversion_tasks`
- `task_retry_logs`
- `system_notifications`
- 第一版统一按 30 天策略清理
